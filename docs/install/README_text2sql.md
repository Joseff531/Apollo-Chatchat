### Apollo-Chatchat Database Conversation Configuration Guide

#### 1. Usage Recommendations

> 1. Because the SQL generated by an LLM may deviate from what you expect, please make sure to test and evaluate thoroughly in a test environment.
> 2. In production environments, for query operations, since query efficiency cannot be guaranteed, we recommend using a primary/replica database architecture and letting text2sql connect to the replica database, to prevent potential slow queries from impacting the main business.
> 3. Write operations should be handled cautiously. If write operations are not required, set `read_only` to `True`, and preferably also revoke write privileges from the database user at the database level to prevent users from modifying the database through natural language.
> 4. The effectiveness of text2sql depends on the LLM's capabilities in intent understanding and SQL conversion. You can switch between different LLMs to test.
> 5. Database table names and field names should be consistent with their actual purpose and easy to understand, and should be annotated in detail to help the LLM better understand the database structure.
> 6. If existing database table names are hard for an LLM to understand, you can configure the `table_comments` field to provide supplementary descriptions of certain tables.

#### 2. Configuration Guide

##### 1. Configuration node
After initialization, locate the `text2sql` configuration node in the `tool_settings.yaml` file:


```yaml
text2sql:
  model_name: qwen-plus
  use: false
  sqlalchemy_connect_str: mysql+pymysql://username:password@host_address/database_name
  read_only: false
  top_k: 50
  return_intermediate_steps: true
  table_names: []
  table_comments: {}
```

##### 2. Explanation of main parameters
1. **model_name**

  This tool requires you to specify the LLM to use separately; it is independent of the model selected on the user-facing frontend.

2. **sqlalchemy_connect_str**

  SQLAlchemy connection string. Supported databases include: crate, duckdb, googlesql, mssql, mysql, mariadb, oracle, postgresql, sqlite, clickhouse, prestodb.

  For different databases, please consult the SQLAlchemy documentation and modify `sqlalchemy_connect_str` to configure the corresponding database connection. For example, sqlite uses `sqlite:///path_to_database_file`.

  If you see a prompt about a missing driver for a database, please install it yourself via poetry.

3. **read_only**

  Setting this to `true` enables read-only mode. However, we still strongly recommend restricting user privileges at the database level first.

4. **top_k**

  Limits the number of rows returned.

5. **table_names**

  If `table_names` is not specified, `SQLDatabaseSequentialChain` is used first. This chain first predicts which tables are needed, and then feeds the relevant tables into `SQLDatabaseChain`. The reason is that without specifying `table_names`, using `SQLDatabaseChain` directly causes Langchain to pass the full table schema to the LLM, which may cause errors due to overly long tokens and wastes resources. However, if there are many tables, `SQLDatabaseSequentialChain` will also consume many tokens.

  If `table_names` is specified, `SQLDatabaseChain` is used directly, passing only the specified table schemas to the LLM for evaluation, which can save resources.
  An example of using specific tables is as follows:

  ```yaml
    table_names: ["sys_user","sys_dept"]
  ```

6. **table_comments**


   If the LLM selects the wrong table, you can try additionally declaring table names and corresponding descriptions based on the actual situation, for example:

  ```yaml
    table_comments: {"tableA":"This is a user table that stores basic user information","tanleB":"Role table"}
  ```
